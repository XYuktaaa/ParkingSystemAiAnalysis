📐 Model Rating & Optimization Roadmap

Based on your pipeline + the Bengaluru BTP dataset characteristics
🗺️
DBSCAN Clusterer
eps=150m · min_samples=10 · haversine metric
7.5 / 10
Current rating
✅ What works: Haversine distance is correct for GPS lat/lon · cluster on only validated+GPS-valid rows · `n_jobs=-1` parallelism.
⚠️ Weaknesses: eps=150m is a fixed guess — Bengaluru arterials can be 500m apart; small eps creates too many noise points. min_samples=10 may miss low-density suburban hotspots. Only `has_junction==1` rows used — the 147 K "No Junction" records are a goldmine you're skipping.
🚀 Optimizations: (1) Auto-tune eps via k-distance elbow plot (k=min_samples). (2) Run a second DBSCAN on the "No Junction" records separately to surface uncoded hotspots — judges will love this. (3) Use HDBSCAN (pip install hdbscan) which handles variable-density clusters far better. (4) Add `congestion_weight` as a sample weight to bias clusters toward high-impact points.
📈
Forecast Model (XGBoost / RandomForest)
Lag features + rolling stats · 80/20 temporal split
6.5 / 10
Current rating
✅ What works: Lag-1/2/7 + rolling 3 & 7 means is solid. Impact score as feature adds domain knowledge. Temporal train/test split (not random) is correct — prevents data leakage.
⚠️ Weaknesses: Simple 80/20 split doesn't test across junctions (a model trained on popular junctions may not generalise to low-frequency ones). No cross-validation — you have no uncertainty estimate on R². Jan has 4× more records than April; the model may learn Jan patterns and perform poorly on sparse months. Missing: holiday/event features, rainfall proxy, month-end effects.
🚀 Optimizations: (1) Use TimeSeriesSplit(n_splits=5) for proper OOF evaluation — you already import it but don't use it. (2) Add Optuna hyperparameter tuning (n_estimators 100-800, max_depth 3-8, lr 0.01-0.3). (3) Add LightGBM + CatBoost to an ensemble — average predictions or stack with Ridge. (4) Add is_holiday flag for Karnataka public holidays. (5) Try target encoding `junction_clean` with smoothing to handle 150 cardinalities. (6) Log-transform target (violation count often right-skewed) → train on log(count+1), predict, exponentiate back.
⚠️
Anomaly Detector (IsolationForest)
contamination=5% · features: count, impact, main_road_frac, z_score
7 / 10
Current rating
✅ What works: Z-score feature gives temporal context. Using `decision_function` score (not just flag) allows ranking anomalies. contamination=0.05 is reasonable for sparse event spikes.
⚠️ Weaknesses: Global model across all junctions — a 5-violation day at a low-traffic junction is different from 5 at Safina Plaza. No per-junction normalisation before fitting. 4 features is thin; no day-of-week or month control.
🚀 Optimizations: (1) Fit per-junction IsolationForest (or at least per cluster-tier: Critical/High/Medium). (2) Add `dow` and `month` as features to avoid flagging Monday-surges as anomalies. (3) Try LocalOutlierFactor — better for density-based anomaly detection in sparse temporal data. (4) Set a rolling alert threshold: flag if daily_count > roll7_mean + 2.5 × roll7_std — simpler and more explainable to judges.
🏆 Overall Pipeline Rating: 7 / 10
What's genuinely good: End-to-end pipeline in one file, congestion proxy scoring is well-designed and defensible, feature engineering is thorough (peak flags, violation type dummies, junction factors), artefact persistence is production-ready.

Critical gaps to fix before submission:
① Empty dataset guard — no checks if CSV is empty or GPS rows = 0 after filtering; DBSCAN will crash. Add: if gps.empty: return empty_df, empty_summary, None
② XGBoost fallback — if `forecast_df` after dropna() has <50 rows (can happen with sparse junctions), model will overfit silently. Add a row-count guard.
③ No TimeSeriesSplit CV — import exists, not used. Run 5-fold TSS, log mean±std R² to metrics JSON.
④ Congestion weight = 1 for unmatched rows — sensible default, but unmatched violations get non-zero weight; consider 0 as default.
⑤ No junction encoding — `junction_clean` string isn't used as model feature (only aggregated stats); add smoothed target encoding for per-junction identity signal.
